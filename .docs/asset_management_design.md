## 1. 目的と要件

### 1.1 主な目的（2つの軸）

| 軸 | 説明 |
|---|------|
| **家計簿軸** | 「今、財布にいくらあるか」「銀行にいくらあるか」を厳密に把握 |
| **経費管理軸** | 確定申告のために事業経費を正確に把握 |

これら**2つの軸を両立**させる設計が必要。

### 1.2 制約と前提

- 外部サービス連携（銀行API等）は**行わない**
- 残高は**手入力**で管理
- 支払いがあると、対応する残高から**自動で減算**したい
- クレジットカードの未払い金は**翌月引き落としまで別管理**
- デビットカードは**即時引き落とし**扱い

---

## 2. 現状の問題点

### 2.1 概念の混乱

現在のデータモデルでは、以下の概念が**独立して存在**している：

```
payment_methods（支払方法）
├── cash, bank, credit, emoney, qr, other
└── settlement_timing: immediate | next_month

accounts（残高管理場所）
├── cash, bank, qr, emoney
└── opening_balance_yen

transactions（取引）
└── payment_method_id → payment_methods
```

**問題**: 
- `支払いが発生` → `どの account から減らすか？` が不明確
- `payment_methods` と `accounts` の関係が定義されていない
- クレジットカード払いで「未払い金」がどこに計上されるか曖昧

### 2.2 ダッシュボードの軸が曖昧

- 現在: Transaction（取引）中心の設計
- 追加要件: Balance（残高）中心の表示も必要

→ **何を「中心」にするかの設計判断が必要**

## 3. 概念モデル（提案）

### 3.1 核心概念の定義

```
┌─────────────────────────────────────────────────────────────┐
│                      資産（Assets）                          │
├──────────────┬──────────────────────────────────────────────┤
│  正の資産    │  現金、銀行口座、電子マネー、QR決済残高        │
│ （プラス）   │  → 使うと減る                                 │
├──────────────┼──────────────────────────────────────────────┤
│  負債        │  クレジットカード未払い金                      │
│ （マイナス） │  → 使うと増える（引き落としで精算）            │
└──────────────┴──────────────────────────────────────────────┘
```

### 3.2 データの流れ（フロー図）

```
【日常の支払い】

1. 現金で支払い
   Transaction → 財布（現金）残高が減る

2. デビットカードで支払い
   Transaction → 銀行口座残高が即時減る

3. クレジットカードで支払い
   Transaction → カード未払い金が増える（残高は変わらない）

4. PayPayで支払い
   Transaction → PayPay残高が減る

【月1回の精算】

5. クレジットカード引き落とし
   Settlement → カード未払い金が減る ＋ 銀行口座残高が減る

【資金移動】

6. 銀行 → 財布（現金引き出し）
   Transfer → 銀行残高が減る ＋ 財布残高が増える

7. 銀行 → PayPay（チャージ）
   Transfer → 銀行残高が減る ＋ PayPay残高が増える
```

## 4. データモデル設計案

### 4.1 案A: payment_methods と accounts を統合

```sql
-- payment_methods を拡張し、残高管理も兼ねる
payment_methods (
  id,
  name,                    -- "財布", "三菱UFJ", "楽天カード", "PayPay"
  type,                    -- cash | bank | credit | debit | emoney | qr
  settlement_timing,       -- immediate | next_month
  current_balance_yen,     -- 現在残高（負債の場合はマイナス or 別フラグ）
  is_liability,            -- true = 負債（クレジット未払い）
  linked_bank_id,          -- クレジットカードの引き落とし先銀行
  user_id
)
```

**メリット**: シンプル、1テーブルで完結  
**デメリット**: 概念が混在（支払方法 ≠ 残高管理場所のケースに対応しにくい）

### 4.2 案B: payment_methods と accounts を分離＋リンク（推奨）

```sql
-- accounts: 残高を持つ「器」
accounts (
  id,
  user_id,
  name,                    -- "財布", "三菱UFJ", "PayPay", "Suica"
  account_type,            -- asset | liability
  asset_category,          -- cash | bank | qr | emoney | credit_payable
  opening_balance_yen,     -- 初期残高
  is_active
)

-- payment_methods: 支払い手段
payment_methods (
  id,
  user_id,
  name,                    -- "現金", "三菱UFJ デビット", "楽天カード"
  type,                    -- cash | bank | credit | debit | emoney | qr
  settlement_timing,       -- immediate | next_month | debit_immediate
  source_account_id,       -- 即時引き落とし用: どの account から減らすか
  liability_account_id,    -- 後払い用: どの account（未払い金）に計上するか
  settlement_account_id,   -- 後払い用: 最終的にどの account から引き落とすか
  is_active
)
```

**メリット**: 
- 概念が明確に分離
- 1つの銀行口座に複数のデビットカードを紐づけ可能
- クレジットカードの複雑なフローに対応可能

**デメリット**: 
- テーブル間のリレーションが複雑
- 初期設定が少し手間

## 5. 支払いタイプ別の動作定義

### 5.1 即時引き落とし（現金、デビット、電子マネー、QR）

```
Transaction 作成時:
  accounts[source_account_id].balance -= transaction.amount_yen
```

| 支払方法 | source_account |
|---------|----------------|
| 現金 | 財布 |
| デビットカード | 銀行口座 |
| PayPay | PayPay残高 |
| Suica | Suica残高 |

### 5.2 後払い（クレジットカード）

```
Transaction 作成時:
  accounts[liability_account_id].balance += transaction.amount_yen  # 未払い金が増える

月1回の引き落とし（Settlement）時:
  accounts[liability_account_id].balance -= settlement.amount_yen   # 未払い金が減る
  accounts[settlement_account_id].balance -= settlement.amount_yen  # 銀行残高が減る
```

| 支払方法 | liability_account | settlement_account |
|---------|-------------------|-------------------|
| 楽天カード | 楽天カード未払い | 楽天銀行 |
| 三井住友カード | 三井住友未払い | 三菱UFJ |

## 7. 残高計算ロジック

### 7.0 設計原則: イベントソーシング的アプローチ

**重要**: 残高は「スナップショット」として保存するのではなく、**イベントから常に計算**する。

```
❌ 避けるべき設計:
   UPDATE accounts SET balance = balance - 1000 WHERE id = ?
   → 支出のたびに残高を直接更新

✅ 採用する設計:
   現在残高 = f(初期残高, 全イベント履歴)
   → 残高は SELECT 時に計算、または VIEW/Materialized View で提供
```

**この設計のメリット**:
| メリット | 説明 |
|---------|------|
| **整合性** | 取引を削除/修正しても、残高は自動的に正しくなる（再計算するだけ） |
| **監査性** | 完全な履歴が残る。「なぜこの残高になったか」を追跡可能 |
| **シンプルさ** | 「巻き戻し」ロジックが不要 |
| **堅牢性** | データ不整合が原理的に発生しない |

### 7.1 資産残高の計算式

**重要**: `opened_on`（残高管理開始日）より前の取引は計算に含めない

```sql
-- 資産（現金、銀行、電子マネー等）の現在残高
現在残高 = 初期残高
         + Σ(収入 Transaction: amount_yen > 0 AND source_account = this AND occurred_on >= opened_on)
         - Σ(支出 Transaction: amount_yen < 0 AND source_account = this AND occurred_on >= opened_on)
         + Σ(振替で受取: to_account = this AND occurred_on >= opened_on)
         - Σ(振替で送金: from_account = this AND occurred_on >= opened_on)
         - Σ(カード精算で引落: settlement_account = this AND settled_on >= opened_on)
         ± Σ(残高調整: account = this AND adjusted_on >= opened_on)
```

**なぜ開始日が必要か**:
- 過去の取引データがあっても、残高管理は「ある時点」から始めたい
- 例: 2026年2月1日に「財布 = 10,000円」から管理開始
- 1月の取引は経費管理には使うが、残高計算には含めない

### 7.2 未払い金の計算式

```sql
-- 負債（クレジットカード未払い金）の現在残高
未払い残高 = Σ(クレジット払い Transaction: liability_account = this)
           - Σ(精算 Settlement: liability_account = this)
```

### 7.3 データ変更時の警告とポリシー

#### 7.3.1 取引の削除・修正

```
Transaction を削除/修正した場合:
  → 残高は自動的に再計算される（巻き戻し不要）
  → UI で「残高が変更されます」という警告を表示
  → 削除/修正の履歴を audit_log に記録（任意）
```

**ポリシー**: 
- 支払いは「支払い」として独立した事実
- 支払方法（payment_method）と残高管理（account）は分離して機能する
- **残高管理なしでも支払い記録は成立する**

#### 7.3.2 マイナス残高の警告

**残高がマイナスになる場合がある**:
- 支出登録が先、収入登録が後になった場合
- 初期残高の設定ミス
- 精算漏れ

```
if 計算残高 < 0:
  表示: ⚠️ 「残高がマイナスになっています」
  対応: ユーザーに確認を促す（残高調整 or 入力漏れチェック）
```

#### 7.3.3 クレジットカード引き落としの警告

```
カードの引き落とし日が近づいた場合:
  引き落とし額 = Σ(未精算の Transaction for this card)
  引き落とし元口座残高 = 計算残高(settlement_account)
  
  if 引き落とし額 > 引き落とし元口座残高:
    表示: ⚠️ 「引き落とし日までに残高が不足する可能性があります」
```

### 7.4 残高の取得方法（実装案）

```sql
-- 案1: 都度計算（シンプル、少量データ向け）
CREATE FUNCTION get_account_balance(account_id UUID) RETURNS BIGINT AS $$
  SELECT opening_balance_yen 
       + COALESCE(income, 0) 
       - COALESCE(expense, 0) 
       + COALESCE(transfer_in, 0) 
       - COALESCE(transfer_out, 0)
       - COALESCE(settlements, 0)
       + COALESCE(adjustments, 0)
  FROM accounts a
  LEFT JOIN (...各サブクエリ...)
  WHERE a.id = account_id;
$$ LANGUAGE SQL;

-- 案2: View（常に最新、JOIN時に計算）
CREATE VIEW account_balances AS
SELECT 
  a.id,
  a.name,
  a.opening_balance_yen + ... AS current_balance_yen
FROM accounts a
LEFT JOIN ...;

-- 案3: Materialized View（パフォーマンス優先、定期更新）
-- 大量データの場合に検討
```

## 10. 決定事項（レビュー結果）

### 10.1 回答済み事項

1. **初期残高の入力タイミング**
   - ✅ **決定**: 口座追加時に都度入力

2. **残高のずれが発生した場合**
   - ✅ **決定**: ユーザー側で「残高調整」機能を使って調整
   - 理由: 外部連携がないため、システム側でずれを検知することは不可能
   - `balance_adjustments` テーブルに記録（理由メモ付き）

3. **クレジットカードの請求額**
   - ✅ **決定**: 自動計算 ＋ 明細CSV取り込み時に検証
   - 自動計算: 対象期間の利用額を合算
   - 検証: CSV取り込み時に、計算値と請求額を比較して差異を表示

4. **収入の扱い**
   - ✅ **決定**: Transaction として記録する
   - `transactions.amount_yen` の符号で区別（正=収入、負=支出）
   - 給与振込: amount_yen > 0, source_account = 銀行口座
   - 現金収入: amount_yen > 0, source_account = 財布

5. **案A/案Bの選択**
   - ✅ **決定**: 案B（payment_methods と accounts を分離＋リンク）
   - 理由: 概念が明確、拡張性が高い、クレジットカードのフローに対応可能

### 10.2 設計ポリシー（確定）

| ポリシー | 説明 |
|---------|------|
| **イベントソーシング** | 残高は直接更新せず、イベント履歴から計算 |
| **分離原則** | 支払い記録と残高管理は独立して機能する |
| **警告重視** | 削除/修正時は警告表示、巻き戻しロジックは実装しない |
| **マイナス許容** | 残高はマイナスになりうる（警告を表示） |
