# Phase 2: 表記ゆれ吸収＋ルール化（3ヶ月後）

## 1. 目的
- ベンダー（取引先）の表記ゆれを吸収し、同一店舗を統一的に扱えるようにする。
- 過去の判定履歴から学習し、判定作業を効率化する。
- 複数の入力経路（カード、口座、d払い、現金）に対応する。

## 2. スコープ

### 2.1 ベンダーマスタ化
- `vendors` テーブルを追加し、正式名称を管理。
- `vendor_aliases` テーブルで表記ゆれ（エイリアス）を管理。
- `transactions.vendor_norm` から `vendors` を引けるようにする。

### 2.2 Smart Suggestions（判定支援）
- 同じvendorの過去の判定履歴を表示。
- 「前回の判定を適用」ボタンでワンクリック適用。
- vendor単位でデフォルトカテゴリを学習。

### 2.3 Batch Actions（一括処理）
- vendor単位で「今後すべて経費100%」などのルールを設定。
- ルール適用履歴を記録。

### 2.4 カテゴリ導入
- `expense_categories` テーブルを追加。
- 自分が管理しやすい粒度でカテゴリを管理（交通費、通信費、消耗品など）。
- `transaction_business_info.category_id` にFK制約を追加。

### 2.5 複数入力経路対応
- 口座明細CSV、d払い明細CSVのインポート対応。
- 現金入力（Googleフォーム → CSV経由での取り込み）。

## 3. 受け入れ基準

### 3.1 DB・スキーマ
- [ ] `vendors`, `vendor_aliases`, `expense_categories` テーブルが作成される。
- [ ] `transaction_business_info.category_id` にFK制約が追加される。

### 3.2 ベンダー管理
- [ ] vendors/vendor_aliasesを手動で登録できる。
- [ ] 表記ゆれ（例：「セブンイレブン」と「セブン-イレブン」）が同一vendorとして扱われる。
- [ ] vendorにデフォルトカテゴリを設定できる。

### 3.3 Smart Suggestions & Batch Actions
- [ ] Triage Queueで過去判定の表示とワンクリック適用ができる。
- [ ] vendor単位で一括判定ルールを設定・適用できる。

### 3.4 カテゴリ & インポート
- [ ] カテゴリ別集計（月次レポート）が表示される。
- [ ] 口座、d払い、現金の各CSVを取り込め、適切な支払い手段が自動選択される。
- [ ] インポート設定（列マッピング）を保存・再利用できる。
