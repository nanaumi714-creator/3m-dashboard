# 📋 全フェーズ要件＆受け入れ基準（完全最終版）

---

## 🎯 プロジェクト全体目的

フリーランスとしての金銭管理・証憑管理を、**Phase 1は課金ゼロ**、Phase 3/4は**必要なら課金許容**で実現する。会計申告の自動化ではなく、**自分の目でお金の流れを把握**し、将来的に必要なら手動で確定申告に使えるデータを準備できる状態を保つ。

# Phase 1: MVP（最小機能・ローカル運用・課金ゼロ）

## 要件

### 目的
- **課金ゼロ**でフリーランスの金銭管理を実現
- **自分の目でお金の流れを把握**できることを第一目的とする
- 将来的に**必要なら手動で確定申告に使える粒度**のデータを保持する

### スコープ

#### 1. 実行環境・制約
- **ローカルでのみ動作**（PC稼働時のみ閲覧可能）
- 環境は**再現性のある手順で起動・停止**できること
- **課金ゼロ前提**（Supabase Cloud、Vercel、OCR API等は使用しない）
- 通貨は**円（JPY）固定**。外貨は扱わない

#### 2. 技術構成
- フロント：**Next.js（ダッシュボードWeb画面）**
- DB：**Supabase Local（Docker経由）**
- Storage：**Supabase Local Storage**
- インポート処理：**Python CLI（Importer）**
- 起動管理：**固定手順（README記載）**、将来的にMakefile化は任意
- ローカル限定アクセス：Web UIは**127.0.0.1にバインド**する

#### 3. 取り込み（入力経路）
- **Phase 1では1種類のカード明細CSVのみ対応**
  - 対応フォーマット例：三井住友カード、楽天カード等（実装時に1つ選定）
  - 列マッピング（日付、金額、摘要）が明確に定義されている
- 半自動取り込み方式（ファイル配置 → CLI実行 → 自動整形・登録）

#### 4. データ管理（DB設計）
Phase 1で作成するテーブル:
- `payment_methods` - 支払い手段マスタ
- `import_sources` - インポート履歴（追跡性）
- `transactions` - 取引本体（円、支出は負／収入は正）
- `transaction_business_info` - 経費判定（別テーブルで監査可能に）
- `receipts` - 証憑（Phase 1では最小実装）

設計原則:
- 経費判定は `transaction_business_info` の存在で管理（nullable地獄回避）
- 追跡性のため `import_sources` にインポート単位の情報を保持
- 重複排除のため `transactions.fingerprint` を保持
- 同一インポートの同一行を二重登録しない制約

#### 5. 取引の重複検出
fingerprint生成材料（SHA256でハッシュ化）:
- occurred_on（取引日）
- amount_yen（金額）
- payment_method_id（支払い手段）
- vendor_norm または description正規化値
- source_type

運用方針:
- fingerprintは**ユニーク制約にしない**（同日同額等の誤判定を避ける）
- 手動マージ用に `duplicate_group_id` を保持
- **Phase 1では重複候補カウント表示まで**（候補一覧UIはPhase 2）

#### 6. 可視化（ダッシュボード）
最低限の閲覧・判断ができる画面:
- **Transactions一覧**（検索・フィルタ・ページング必須）
- **Triage Queue**（未判定の経費判定を処理）
- **月次集計ダッシュボード**

主要な可視化観点:
- 月次の収入／支出（支出中心でOK）
- 事業支出（按分適用後）
- 支払い手段別の支出傾向
- 未判定件数

#### 7. 証憑管理（Phase 1段階的実装）
- **Phase 1最小**：receiptsテーブルへのレコード登録、storage_url表示
- **Phase 1完成**：Web UIからのアップロード、Supabase Local Storageへの保存

#### 8. 非機能要件
- **冪等性**：同じCSVを複数回インポートしてもDBが壊れない
- **パフォーマンス**：3ヶ月分（100-150件）で実用速度（ページング実装）
- **バックアップ**：DBバックアップ/リストア手順がREADMEに記載

# Phase 2: 表記ゆれ吸収＋ルール化（3ヶ月後）

## 要件

### 目的
- ベンダー（取引先）の表記ゆれを吸収し、同一店舗を統一的に扱えるようにする
- 過去の判定履歴から学習し、判定作業を効率化する
- 複数の入力経路（カード、口座、d払い、現金）に対応する
- 証憑管理の完成（アップロードUI、重複候補管理UI）

### スコープ

#### 1. ベンダーマスタ化
- `vendors` テーブルを追加し、正式名称を管理
- `vendor_aliases` テーブルで表記ゆれ（エイリアス）を管理
- `transactions.vendor_norm` から `vendors` を引けるようにする
- 既存の `transactions` データを `vendors` に紐付ける移行スクリプト

#### 2. Smart Suggestions（判定支援）
- 同じvendorの過去の判定履歴を表示
- 「前回の判定を適用」ボタンでワンクリック適用
- vendor単位でデフォルトカテゴリを学習
- 判定候補の信頼度スコア表示（過去の一貫性から算出）

#### 3. Batch Actions（一括処理）
- vendor単位で「今後すべて経費100%」などのルールを設定
- `vendor_rules` テーブルでルールを永続化
- ルール適用履歴を記録（audit log）
- ルールの有効/無効切り替え

#### 4. カテゴリ導入
- `expense_categories` テーブルを追加
- 自分が管理しやすい粒度でカテゴリを管理（交通費、通信費、消耗品など）
- `transaction_business_info.category_id` にFK制約を追加
- カテゴリのソート順管理
- カテゴリの有効/無効フラグ

#### 5. 複数入力経路対応
- 口座明細CSVのインポート対応
  - 列マッピング設定の保存・再利用
  - 入出金の自動判定（入金=正、出金=負）
- d払い明細CSVのインポート対応
  - ポイント利用の扱い（任意）
- 現金入力（手動入力UIまたはGoogleフォーム経由CSV）
  - 簡易入力フォーム（日付・金額・説明のみ）
- インポート設定の保存・管理
  - `import_configs` テーブル（source_type, column_mapping等）

#### 6. 証憑アップロードUI（Phase 1完成版の強化）
- Web UIからの画像/PDFアップロード
- Supabase Storageへの自動保存
- 取引への紐付け（ドラッグ＆ドロップ推奨）
- サムネイル表示
- ファイルサイズ制限（10MB推奨）

#### 7. 重複候補一覧UI
- 重複候補の一覧表示（fingerprintが一致する取引）
- 手動マージ機能（`duplicate_group_id` 設定）
- マージ履歴の記録
- マージの取り消し機能

# Phase 3: 証憑管理強化＋データ出力（必要になったら・課金許容）

## 要件

### 目的
- 証憑（レシート・領収書）の管理を強化する
- 必要に応じて外部ツール（Excel、会計ソフト）にデータを出力できるようにする
- **課金許容**（OCR API、追加ストレージ等）

### スコープ

#### 1. OCR連携
- OCR APIによる自動テキスト抽出（ocr_text, ocr_confidence）
- 金額・日付の自動抽出と取引への紐付け候補表示
- OCR結果の手動修正機能
- **課金許容**（Google Cloud Vision API、AWS Textract等）
- OCR精度のフィードバック機能（学習データとして活用）

#### 2. データエクスポート機能
- 汎用CSV出力（期間指定、カラム選択可能）
- Excel形式出力（.xlsx）
- 出力テンプレートの保存・再利用
- エクスポート履歴の記録
- カスタムフォーマット定義（JSON設定）

#### 3. メール領収書取り込み
- Gmailから領収書PDFを自動検出
  - Gmail API連携（OAuth認証）
  - 検索クエリの保存・再利用
- 添付ファイルをSupabase Storageに保存
- receiptsテーブルに自動登録
- メール本文から金額・日付を抽出
- 定期自動取り込み（cron設定）

#### 4. 高度な検索・分析
- 全文検索（description, vendor, ocr_text）
- 複雑な条件での絞り込み
  - 金額範囲
  - 複数カテゴリ
  - AND/OR条件
- 保存された検索条件の再利用
- カスタムレポート作成
  - 期間比較（前月比、前年比）
  - カテゴリ別トレンド
